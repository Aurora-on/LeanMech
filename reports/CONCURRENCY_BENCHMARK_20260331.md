# 题目级并发基准报告

日期：2026-03-31

## 目的

验证新引入的 `sample_concurrency` 是否能在真实 LLM 请求下缩短整次 pipeline run 的墙钟时间。

本次只比较总耗时，不比较不同 run 之间的证明质量优劣。真实 LLM 输出存在波动，因此质量指标只能作为附带现象，不能作为本报告的主结论。

## 测试设置

- 固定配置：`configs/mechanics73_plus3_proxy_gpt54_seed2026033001.yaml`
- 固定题集：`fixtures/bench_mechanics73_plus3_seed2026033001.json`
- 模型：`gpt-5.4`
- 请求方式：真实 API 调用，使用 OpenAI-compatible 代理端点 `https://api.openai-proxy.org/v1`
- 计时口径：从执行 `python -m mech_pipeline.cli run ...` 开始，到进程退出结束的总墙钟时间
- 对比变量：
  - `--limit`
  - `--sample-concurrency`

固定 bench 中的 4 题为：

1. `university_mechanics_Mechanics_73_University`
2. `university_mechanics_Mechanics_6_University_Significant_figures_in_multiplication`
3. `university_mechanics_Mechanics_34_University_copy`
4. `competition_mechanics_Ch1_Q2`

- 串行：`295.17s`

## 结论

结论是明确的：题目级并发在真实 API 请求下确实能显著减少总运行时间，而且题量越大，收益越明显。

- `2` 题时已经出现可见收益，但幅度有限。
- `4` 题时收益明显，`sample_concurrency=4` 相比串行将总耗时从 `866.42s` 降到 `414.51s`。
- 以当前这组 workload 看，题目数接近并发数时，吞吐提升最明显。

## 解释与边界

- 这次测的是整条 pipeline 的总墙钟时间，不只是 LLM 请求时间；因此收益同时包含了 LLM 调用重叠和 Lean/本地处理重叠。
- 加速没有达到理论上的线性倍数，这是正常现象。原因包括：
  - 单题内部仍然是串行链路
  - Lean 编译与 proof 检查也占用时间
  - 网络波动和模型端排队时间不可控
  - 不同题目本身难度不同，最慢样本会拖住整批完成时间
- 由于是真实 LLM 调用，不同 run 的最终指标会波动。例如 `4` 题串行与 `4` 题并发 `4` 的 `metrics.json` 并不完全一致。这不影响本报告关于“总耗时是否减少”的结论，但意味着如果要进一步做“速度/质量联合评估”，需要追加多轮重复实验。

## 建议

- 日常 2 至 4 题小批量运行时，可将 `sample_concurrency` 设为与题量相同或略低。
- 如果后续要跑更大的批量，建议继续记录：
  - 总耗时
  - 每题耗时分布
  - API 错误率
  - proof 成功率波动
- 如果目标是继续优化吞吐，下一个最有价值的方向不是再堆更多全局分支，而是：
  - 给不同 provider 加限流配置
  - 拆分 LLM 阶段和 Lean 阶段的并发上限
  - 记录每题各阶段耗时，定位真正的长尾瓶颈
